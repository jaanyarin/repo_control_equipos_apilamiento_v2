## 3. Actualización de Plataforma (Node.js + Expo SDK)

Previo a la validación del login mobile, se realizó una actualización de plataforma para compatibilidad con Expo Go v54 instalado en el dispositivo de prueba:

| Componente | Versión Anterior | Versión Actual |
|---|---|---|
| Node.js | v20.9.0 | v22.22.3 LTS |
| Expo SDK | 52 | 54 |
| React Native | 0.76.x (SDK 52) | 0.76.7 (SDK 54) |
| expo-web-browser | ~13.x | ~14.0.2 |
| expo-auth-session | ~5.x | ~6.0.0 |
| expo-secure-store | ~13.x | ~14.0.0 |

**Motivo**: El dispositivo de prueba (realme C21Y) tenía Expo Go v54 instalado, incompatible con SDK 52. Se optó por **Opción B** (actualizar Node.js + proyecto a SDK 54) en lugar de instalar Expo Go v52, para evitar tener que actualizar nuevamente al momento de compilar el APK final.

**Procedimiento**:
1. Node.js v20.9.0 actualizado a v22.22.3 (vía ZIP manual en `%USERPROFILE%\nodejs22\node-v22.22.3-win-x64`)
2. Se eliminó `node_modules` y `package-lock.json`
3. `npm install` con Node.js v22 — 873 paquetes instalados sin errores
4. Expo CLI v54.0.24, SDK del proyecto v54.0.0
5. Metro Bundler compila correctamente

## 4. Funcionalidades Verificadas

### 4.1 Infraestructura
- Contenedores Docker: PostgreSQL 18, Backend Quarkus, Nginx
- PostgreSQL con timezone America/Lima (UTC-5)
- Migraciones Flyway aplicadas hasta V3
- Frontend web SPA servido por Nginx, reverse proxy `/api/` al backend
- Backend accesible en `http://localhost/api/v1/`

### 4.2 Autenticación (MOD-01)
- Login con cuenta Microsoft corporativa vía OIDC (flujo authorization_code)
- Validación de dominio corporativo
- Validación de usuario registrado en dim_usuarios
- Validación de cuenta activa en Microsoft Graph API
- Validación de estado_activo en dim_usuarios
- JWT con claims: sub, upn, groups, nombre, correo, rolId
- Cierre de sesión con hard redirect (`?logout=1`)
- Manejo de errores de autenticación (`?error=`)
- Prevención de auto-login en logout/error

### 4.3 Usuarios (MOD-02)
- Listado de usuarios con datos corporativos sincronizados
- Creación de usuario con autocompletado de correo Microsoft Graph
- Edición de rol y estado activo
- Eliminación física con diálogo de confirmación
- Filtrado de roles según el rol del usuario logueado
- Roles con colores: Super Admin (verde), Admin (azul), Usuario (amarillo)

### 4.4 Sedes (MOD-03)
- CRUD completo (Crear, Listar, Actualizar, Eliminar)
- Un solo campo de entrada: "Sede" (nombre)
- Código auto-generado desde el nombre (mayúsculas, espacios → guion bajo, sin caracteres especiales)
- Eliminación física con confirmación (no soft delete)
- Validación de nombre único

### 4.5 Campañas (MOD-04)
- CRUD completo (Crear, Listar, Actualizar, Eliminar)
- Un solo campo de entrada: "Campaña" (nombre)
- Campos adicionales: fecha_inicio, fecha_fin
- Código auto-generado desde el nombre
- Solo una campaña activa a la vez
- Botones Activar/Cerrar por fila
- Fila activa resaltada en verde
- Eliminación física con confirmación (no soft delete)
- Estado mostrado como Chip: "Activa" (verde) / "Cerrada" (gris)

### 4.6 Mobile App (MOD-16)
- Login OIDC mediante WebBrowser nativo (expo-web-browser)
- Pantalla de ingreso exitoso: "Ingresaste de forma correcta" + nombre del usuario + correo
- Persistencia de sesión con SecureStore (token JWT)
- Botón "Cerrar sesión" que elimina el token y regresa al login
- Recuperación automática de sesión al abrir la app
- URL del backend configurable via variable de entorno `EXPO_PUBLIC_API_URL`
- Deep link callback (`exp://.../--/callback`) para recibir el token post-autenticación
- Sin navegación a Dashboard — pantalla de bienvenida como placeholder temporal
- Framework: Expo SDK 54 + React Native Paper + React Navigation

## 6. Decisiones Técnicas Validadas

| Decisión | Detalle |
|---|---|
| Soft delete | Datos operacionales → soft delete (estado_activo). Excepciones: usuarios, sedes, campañas → DELETE físico |
| Timezone | PostgreSQL configurado con `TZ=America/Lima` + `-c timezone=America/Lima`. Backend usa `OffsetDateTime.now(ZoneId.of("America/Lima"))` |
| Auto-generación de código | `nombre.toUpperCase().replaceAll("\\s+", "_").replaceAll("[^A-Z0-9_]", "")` |
| Campaña activa única | Al activar una campaña se desactiva automáticamente la anterior |
| Auth flow | Backend realiza fetch manual a Microsoft Entra ID (HttpClient), genera JWT propio |
| Frontend web serving | Vite build → nginx serve static files with SPA routing |
| Frontend mobile | Expo SDK 54 con React Native Paper, SecureStore, WebBrowser auth |
| API URL mobile | Configurable via `EXPO_PUBLIC_API_URL` en `.env` o variable de entorno |
| Logout web | Hard redirect con `window.location.href` + `?logout=1` |
| Logout mobile | Eliminación de token de SecureStore + reset de estado |
| Error handling | Backend redirige a `/?error=<mensaje>`, frontend muestra Alert |

## 8. Instrucciones para Retomar el Desarrollo

Al retomar el desarrollo desde este hito:

1. **No modificar** los archivos de los módulos validados (sedes, campañas, usuarios, roles, auth, mobile login) a menos que sea estrictamente necesario por integración.
2. **Iniciar por HDT-002**: Tipos de Equipo, Proveedores, Equipos.
3. **Seguir el patrón** establecido por Sedes/Campañas:
   - Backend: Entity → Repository → Service → Resource (JAX-RS) + DTO + Mapper
   - Frontend web: Página en `frontend/src/pages/`, ruta en `App.jsx`, menú en `Layout.jsx`
   - Frontend mobile: Pantalla en `mobile/src/`, ruta en `App.js`
   - DB: Migración Flyway (`V4__*.sql`)
4. **Documentar** cada nuevo módulo en el próximo hito (`05_hito_002.md`).
5. **Para desplegar cambios web**: `npm run build` en `frontend/` y `docker compose up -d --build`
6. **Para mobile**: `cd mobile && npx expo start` y escanear QR con Expo Go
7. **Para cambiar de WiFi**: editar `mobile/.env` con la nueva IP y reiniciar Expo
8. **Verificar** que los endpoints retornen 401 sin token y 200/201/204 con token válido.

## 10. Notas Adicionales

- Las campañas y sedes existentes en la base de datos antes de este hito no tienen migración de datos asociada; se crean desde cero.
- La eliminación física de sedes fallará si hay usuarios con `sitio_id` referenciando esa sede (FK constraint). Es el comportamiento esperado.
- La eliminación física de campañas no tiene restricciones actualmente (FK no implementadas aún en tablas futuras).
- La aplicación mobile usa Expo Go para desarrollo; para APK final se requerirá compilar con `eas build`.
- El deep link `exp://.../--/callback` es manejado automáticamente por Expo Linking.
